# 参考图解析提示词设计文档

- 日期：2026-03-02
- 项目：`cine`（Vite + React + TypeScript）
- 目标：在生成界面上传参考图后，自动调用 Gemini 分析图像并回填英文详细提示词。

## 1. 背景与问题

当前 `src/App.tsx` 的上传逻辑是本地模拟：

- 上传后仅通过 `setTimeout` 模拟“分析中”
- 固定写死一段提示词
- 未接入真实模型分析能力

需要改造为真实可用流程，并确保 API Key 不暴露到前端。

## 2. 已确认需求

1. 上传后结果写入方式：直接覆盖提示词输入框。
2. 调用方式：前端走后端代理，后端调用 Gemini。
3. 现有提示词处理：上传分析成功后覆盖旧内容。
4. 输出语言：英文。
5. 模型选择：`gemini-3-flash-preview`。
6. 触发方式：上传后自动立即分析。
7. 输出风格：详细型提示词（非精简）。
8. 失败处理：提示失败并保留原提示词不变。

## 3. 方案对比

### 方案 A：前端直连 Gemini（不采用）

- 优点：接入快，改动少。
- 缺点：密钥暴露风险高，不满足安全要求。

### 方案 B：同仓库新增轻量后端代理（采用）

- 优点：安全性更好；改动范围可控；便于后续扩展限流与日志。
- 缺点：需要新增后端入口及开发流程。

### 方案 C：拆分独立后端服务（暂不采用）

- 优点：边界最清晰，扩展性最强。
- 缺点：当前体量下实施成本偏高。

## 4. 架构设计

### 4.1 模块划分

- 前端：`src/App.tsx`
  - 负责文件选择、上传、状态展示、结果覆盖。
- 后端：`server/`（新增）
  - 提供 `POST /api/image-to-prompt`。
  - 服务端读取 `GEMINI_API_KEY` 并调用 `@google/genai`。

### 4.2 责任边界

- 前端不直连模型、不读取密钥。
- 后端统一封装模型调用与参数校验。

## 5. 数据流与交互

1. 用户选择参考图。
2. 前端进入 `analyzing` 状态，显示“分析中”。
3. 前端将图片以 `multipart/form-data` 发到后端接口。
4. 后端调用 `gemini-3-flash-preview` 输出详细英文提示词。
5. 前端成功后覆盖提示词输入框并结束加载。
6. 若失败，提示“分析失败，请重试”，并保持原提示词不变。

### 并发策略

- 快速连续上传时，只以“最后一次上传”的结果为准，避免竞态覆盖。

## 6. 接口设计

### 6.1 `POST /api/image-to-prompt`

- `Content-Type`：`multipart/form-data`
- 入参：
  - `file`：单张图片（`image/jpeg`、`image/png`、`image/webp`）
- 约束：
  - 大小限制（建议 10MB）
- 成功返回：

```json
{
  "prompt": "detailed english prompt...",
  "model": "gemini-3-flash-preview"
}
```

- 失败返回：

```json
{
  "error": "error message"
}
```

## 7. 提示词生成策略

- 模型：`gemini-3-flash-preview`
- 输出要求：
  - 仅返回英文提示词纯文本
  - 不要解释说明、不要 markdown
  - 风格为详细型（主体、场景、构图、镜头、光线、色调、细节、风格标签、质量标签）
- 安全约束：
  - 避免输出敏感或侵权引导内容

## 8. 错误处理策略

### 前端

- 非 2xx：结束加载，提示失败，保留原提示词。
- 超时或中断：按失败提示，可重试。

### 后端

- 明确区分参数错误与模型调用失败：
  - `400`：缺少文件、格式错误、超大小
  - `500`：模型调用异常
- 不回传敏感信息（密钥、堆栈）。

## 9. 验收标准

1. 前端不再直接调用 Gemini。
2. 后端真实调用 `gemini-3-flash-preview`。
3. 上传后自动分析并覆盖提示词输入框。
4. 分析失败时保留原提示词并可重试。

## 10. 验证计划

1. `npm run lint` 通过。
2. `npm run build` 通过。
3. 手工验证：
   - 正常上传成功路径
   - 接口失败路径
   - 连续上传竞态路径
   - 非法文件/超大小路径
