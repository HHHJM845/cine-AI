# 即时交互式反馈（批次级）设计

## 1. 背景与目标
当前生成页已支持按批次展示文生图结果，但缺少结构化的用户反馈入口，不利于后续 Badcase 归因分析。  
本次目标是在每组生成结果（batch）下提供即时反馈能力，并将反馈持久化，保证刷新后可回显、可迭代覆盖。

目标包括：
- 在每个批次下支持 `点赞/点踩` 与文本意见提交。
- 点踩时弹出二级浮窗，要求选择至少一个负反馈原因，并支持补充文本。
- 反馈按批次持久化存储，同一批次可反复修改，始终覆盖为最新值。
- 为后续 Badcase 分析提供可结构化消费的数据。

## 2. 需求冻结（已确认）
- 反馈粒度：按整组批次（batch），不是单图（item）。
- 持久化要求：刷新后仍在，可用于后续 Badcase 分析。
- 反馈能力：
  - 保留点赞/点踩。
  - 去掉五星评分。
  - 支持文本框提交具体意见。
- 点踩追问：
  - 二级浮窗。
  - 固定枚举可多选 + 可选补充文本。
  - 至少选择 1 个原因才能提交点踩。
- 更新策略：同一批次反馈可反复修改，并覆盖为最新一条。

## 3. 方案对比与选择

### 方案 A（采用）：批次反馈单表 + 覆盖式 Upsert
- 设计：新增 `generation_batch_feedback`，`batch_id` 唯一；每次提交对同一批次执行 upsert。
- 优点：
  - 与“覆盖最新态”需求完全一致。
  - 查询与前端合并逻辑简单。
  - 改造范围可控，风险最低。
- 缺点：
  - 不保留修改历史。

### 方案 B（不采用）：事件流追加写入 + 读取最新
- 优点：可追踪反馈演进。
- 缺点：实现复杂度与查询成本更高，超出当前需求。

### 方案 C（不采用）：直接扩展 `generation_batches`
- 优点：表少、读取直接。
- 缺点：反馈模型与生成模型耦合过重，后续扩展受限。

## 4. 架构与数据模型设计

### 4.1 新增表
表名：`generation_batch_feedback`

建议字段：
- `batch_id TEXT PRIMARY KEY`
- `vote TEXT`（允许 `up/down/null`）
- `downvote_reasons TEXT NOT NULL DEFAULT '[]'`（JSON 字符串数组）
- `comment TEXT NOT NULL DEFAULT ''`
- `created_at INTEGER NOT NULL`
- `updated_at INTEGER NOT NULL`

约束规则：
- 当 `vote = 'down'` 时，`downvote_reasons` 必须至少包含 1 项。
- 当 `vote != 'down'` 时，`downvote_reasons` 统一落库为 `[]`。
- `comment` 最大长度 500。

### 4.2 点踩原因枚举（多选）
- `风格不符`
- `主体错误`
- `构图问题`
- `细节瑕疵`
- `清晰度不足`
- `与提示词不一致`

### 4.3 前端类型挂载
在 `UiBatch` 上新增 `feedback`（可空）：
- `vote: 'up' | 'down' | null`
- `downvoteReasons: string[]`
- `comment: string`
- `updatedAt: number`

## 5. 交互与组件设计

### 5.1 位置
在生成页每个批次卡片的 Footer Actions 区域新增反馈模块，不影响现有“重新编辑/再次生成/更多”。

### 5.2 主交互
- 点赞按钮：单击设置 `vote='up'`，再次点击可取消为 `null`。
- 点踩按钮：单击打开二级浮窗；只有浮窗内校验通过并提交后才写入 `vote='down'`。
- 文本意见：支持单独输入并提交（不依赖赞踩）。
- 保存反馈：提交当前批次反馈，覆盖旧反馈。

### 5.3 点踩二级浮窗
- 多选原因（枚举）。
- 补充文本（可选）。
- 校验：至少 1 个原因。
- 提交成功后关闭浮窗并更新当前批次反馈展示。

### 5.4 状态反馈
- 已点赞/已点踩按钮高亮。
- 有文本意见时显示“已填写意见”提示。
- 保存中禁用当前批次反馈操作，防止重复提交。

## 6. API 与数据流设计

### 6.1 新增接口

1) `POST /api/generation-batch-feedback`
- 用途：创建或覆盖某批次反馈。
- 请求体：
  - `batchId: string`
  - `vote: 'up' | 'down' | null`
  - `downvoteReasons: string[]`
  - `comment: string`
- 校验：
  - `batchId` 必填且必须存在。
  - `vote='down'` 时 `downvoteReasons.length >= 1`。
  - `vote!='down'` 时服务端强制清空 `downvoteReasons` 为 `[]`。
  - `comment.length <= 500`。
- 返回：最新反馈对象。

2) `GET /api/generation-batch-feedback?batchIds=...`
- 用途：批量拉取当前列表批次对应反馈。
- 返回：`feedbacks` 数组（含 `batchId/vote/downvoteReasons/comment/createdAt/updatedAt`）。

### 6.2 前端数据流
- 页面初始化：
  1. 拉取 `generation-batches`。
  2. 取批次 id 列表调用反馈查询接口。
  3. 按 `batchId` 合并反馈到 `generationBatches`。
- 用户提交反馈：
  1. 前端先本地校验。
  2. 调用 `POST` upsert。
  3. 成功后更新对应 `UiBatch.feedback`。
- 重新生成得到新 `batchId` 时，反馈默认空，不继承旧批次。

## 7. 错误处理与验证

### 7.1 前端校验
- 点踩提交必须至少选择一个原因。
- 文本最长 500，超长即提示并禁用提交。
- 保存中禁用当前批次相关按钮。

### 7.2 后端校验
- 参数合法性：`batchId/vote/downvoteReasons/comment`。
- 业务合法性：点踩必须有原因、批次必须存在。
- 异常返回：`400/404/500`，错误文案可直接透传到前端提示。

### 7.3 用户可见反馈
- 成功：`反馈已保存`。
- 失败：`保存失败：xxx`，保留当前输入并允许重试。
- 初次加载反馈失败：不阻塞批次渲染，仅提示“反馈加载失败，可稍后重试”。

## 8. 测试策略
当前仓库以 Vitest + 手工验证为主，建议最小闭环：

- 后端测试：
  - 路由：upsert 成功、点踩缺原因失败、非法 batchId、批量查询。
  - 仓储：首次写入、重复提交覆盖。
- 前端测试：
  - service：`POST/GET` 成功与失败分支。
  - 手工：点赞、点踩浮窗校验、文本提交、覆盖保存、刷新回显。

## 9. 范围与非目标

### 9.1 范围
- 生成页批次级反馈 UI。
- 点踩二级浮窗与枚举多选。
- 反馈持久化表与读写 API。
- 前端回填与覆盖更新。

### 9.2 非目标
- 不做反馈历史版本追踪。
- 不做用户账号维度权限控制。
- 不做分析看板与报表。
- 不改现有生成队列与主链路。

## 10. 验收标准
- 刷新后反馈可回显。
- 点踩必须选原因才能提交。
- 同批次多次提交后以最新反馈为准。
- 新生成批次（新 `batchId`）反馈不串批次。
