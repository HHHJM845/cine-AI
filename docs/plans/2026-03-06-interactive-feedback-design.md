# 即时交互式反馈（批次级）设计

## 1. 背景与目标
当前生成页按批次展示文生图结果，但缺少结构化反馈入口，导致 Badcase 归因数据不足。  
本次目标是在每组结果（batch）下提供可持久化的互动反馈能力，并在点踩场景下补充二级追问，支持后续精准归因分析。

目标包括：
- 在每个批次 Footer 中，于“再次生成”右侧新增“点赞/点踩”按钮。
- 点击“点踩”时弹出二级浮窗，要求选择至少一个负反馈原因，并支持补充文本。
- 反馈持久化存储，刷新后可回显。
- 同一批次反馈可重复修改，按最新内容覆盖旧内容。

## 2. 需求冻结（已确认）
- 反馈粒度：批次级（batch），非单图级（item）。
- 交互位置：每个批次卡片底部操作区，“再次生成”右侧。
- 点踩追问：二级浮窗，固定 6 个可多选原因，至少选择 1 个才能保存。
- 原因枚举：
  - 风格不符
  - 主体错误
  - 构图问题
  - 细节瑕疵
  - 清晰度不足
  - 与提示词不一致
- 需要持久化：前后端联动，支持刷新回显，用于 Badcase 分析。
- 更新策略：覆盖式更新（保留最新态）。

## 3. 方案对比与选择

### 方案 A（采用）：反馈单表 + `batch_id` 唯一 + 覆盖式 upsert
- 优点：
  - 与“保留最新态”需求完全一致。
  - 查询与前端合并逻辑简单，改造风险低。
  - 易于快速落地并验证。
- 缺点：
  - 不记录反馈修改历史。

### 方案 B（不采用）：事件流追加写入 + 读取最新
- 优点：可追踪反馈演进。
- 缺点：实现复杂度和查询成本上升，超出当前需求。

### 方案 C（不采用）：直接扩展 `generation_batches`
- 优点：表数量更少。
- 缺点：反馈模型与生成模型耦合过重，后续扩展受限。

结论：采用方案 A。

## 4. 架构与数据模型

### 4.1 新增表
表名：`generation_batch_feedback`

建议字段：
- `batch_id TEXT PRIMARY KEY`
- `vote TEXT`（`up/down/null`）
- `downvote_reasons TEXT NOT NULL DEFAULT '[]'`（JSON 字符串数组）
- `comment TEXT NOT NULL DEFAULT ''`
- `created_at INTEGER NOT NULL`
- `updated_at INTEGER NOT NULL`

### 4.2 规则约束
- `vote='down'` 时，`downvote_reasons` 至少 1 项。
- `vote!='down'` 时，服务端强制落库 `downvote_reasons=[]`。
- `comment.length <= 500`。

### 4.3 前端模型扩展
在 `UiBatch` 上新增 `feedback`（可空），包含：
- `vote: 'up' | 'down' | null`
- `downvoteReasons: string[]`
- `comment: string`
- `updatedAt: number`

## 5. 前端交互设计

### 5.1 批次 Footer 操作区
在“重新编辑 / 再次生成”旁新增：
- `点赞` 按钮
- `点踩` 按钮
- `保存反馈` 按钮（提交当前批次草稿）

### 5.2 主交互
- 点赞：单击设为 `up`，再次点击可取消为 `null`。
- 点踩：单击打开二级浮窗，不立即提交。
- 文本意见：支持输入补充描述，作为反馈一部分提交。

### 5.3 点踩二级浮窗
- 标题：`点踩原因（至少 1 项）`
- 内容：6 个固定原因多选 + 补充文本（可选）
- 校验：至少选择 1 项
- 确认后将草稿态标记为 `vote='down'` 并关闭浮窗

### 5.4 状态反馈
- 选中态高亮：已点赞/已点踩按钮有明显状态。
- 保存中禁用当前批次反馈相关按钮，防止重复提交。
- 成功提示：`反馈已保存`；失败提示：`保存失败：xxx`。

## 6. API 与数据流

### 6.1 接口设计
1) `POST /api/generation-batch-feedback`
- 请求体：
  - `batchId: string`
  - `vote: 'up' | 'down' | null`
  - `downvoteReasons: string[]`
  - `comment: string`
- 行为：按 `batchId` upsert（覆盖最新态）
- 返回：`feedback`

2) `GET /api/generation-batch-feedback?batchIds=...`
- 行为：按批次 ID 列表批量查询反馈
- 返回：`feedbacks[]`

### 6.2 前端数据流
- 初始化：
  1. 拉取 `generation-batches`
  2. 取 `batchIds` 调用反馈查询接口
  3. 以 `batchId` 合并反馈到批次列表
- 提交：
  1. 本地校验
  2. 调用 `POST` 保存
  3. 用响应结果更新对应批次反馈并刷新草稿

## 7. 错误处理与校验

### 7.1 前端
- 点踩未选原因：阻止提交并提示 `请选择至少一个点踩原因`。
- 文本超长：阻止提交并提示。

### 7.2 后端
- 参数非法返回 `400`。
- `batchId` 不存在返回 `404`。
- 其他异常返回 `500`。

### 7.3 失败恢复
- 保存失败后保留用户当前草稿，允许直接重试。
- 反馈加载失败不阻塞批次渲染，仅提示可稍后重试。

## 8. 测试策略
- 后端：
  - 请求校验单测（vote 合法性、点踩原因必填、comment 长度）
  - 仓储单测（首次写入、覆盖更新、非 down 原因归零）
  - 路由单测（POST/GET 成功与失败分支）
- 前端：
  - service 单测（POST/GET 正常与错误分支）
  - 手工验证（点赞、点踩追问、保存回显、重复覆盖）

## 9. 范围与非目标

### 9.1 范围
- 批次级点赞/点踩交互
- 点踩二级追问浮窗
- 反馈持久化与回显

### 9.2 非目标
- 不做反馈历史版本追踪
- 不做分析看板
- 不改生成主链路与队列策略

## 10. 验收标准
- 每个批次“再次生成”右侧可见“点赞/点踩”按钮。
- 点踩时必须先选择至少一个原因才能保存。
- 反馈刷新后可回显。
- 同一批次多次提交后，以最新反馈为准。
