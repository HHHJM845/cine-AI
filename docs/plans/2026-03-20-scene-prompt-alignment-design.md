# 场景标签 Prompt 对齐优化设计

## 1. 背景与问题

当前项目已具备完整的场景辅助链路：

- 前端在 `src/App.tsx` 提供一级/二级场景标签选择。
- 后端通过 `server/prompts/<primary>/<sub>.md` 读取场景预设。
- `server/services/scene-assisted-prompt.ts` 将场景预设与用户输入交给 prompt merge 服务合并。

现阶段的主要问题有两类：

1. 二级场景预设虽然拆成独立文件，但内容偏通用创作建议，缺少“影视用途语义”和“视觉语言边界”，导致相邻标签之间差异不够大。
2. prompt merge 层当前更偏“去冲突 + 合并”，对“保留用户主体、强化场景标签语境”的约束不够明确；同时 `prompt-merge-instruction.ts` 与 provider 侧拼接文本存在乱码迹象，会直接影响最终合并质量。

本次设计目标不是单纯把 prompt 写得更长，而是让最终 prompt 在以下维度上更贴合所选场景标签：

- 影视用途语义明确
- 视觉语言明显贴合标签
- 在相邻标签之间保持足够区分度
- 保留用户原始主体、动作、叙事核心

## 2. 需求结论

经确认，本次优化采用以下原则：

- 场景标签强度：`平衡约束`
  - 场景标签负责建立语境、风格和用途。
  - 用户输入中的主体、动作、关系、叙事意图不得被覆盖。
- 问题范围：`预设内容 + 合并规则` 同时调整。
- 成功标准：优先提升
  - 场景语义贴合度
  - 视觉语言贴合度
  - 在此基础上兼顾生成稳定性
- 预设形态：`混合式`
  - 预设文件内部采用结构化表达，便于维护。
  - 最终输出给生图模型的仍是一段自然语言 prompt。

## 3. 方案对比

### 方案 A：仅重写场景预设

- 只调整 `server/prompts/*/*.md`
- 保持现有 merge 指令和 provider 文本拼接逻辑

优点：

- 改动面最小
- 维护成本低

缺点：

- merge 层依然可能稀释场景信息
- 无法解决乱码对结果的污染

### 方案 B：仅重写合并规则

- 只调整 `prompt-merge-instruction` 和 provider 拼接逻辑
- 保留现有场景预设内容

优点：

- 可以统一强化“用户主体优先 + 场景语境增强”的合并逻辑
- 不需要批量重写 15 个场景文件

缺点：

- 预设内容缺少辨识度时，模型没有足够的标签特征可提取

### 方案 C：预设重构 + 合并规则重写

- 重写所有二级场景预设
- 重写 merge 规则
- 修复 merge 指令与输入文本乱码

优点：

- 标签差异由预设提供
- 输出稳定性由 merge 规则保证
- 最符合“贴标签但不改题”的目标

缺点：

- 工作量最大

推荐采用 `方案 C`。

## 4. 设计方案

### 4.1 场景预设文件结构

每个二级场景预设文件从“泛化说明”升级为“结构化场景语义卡”，统一包含以下信息：

- `核心用途`
  - 说明该标签在影视生产或宣发中的真实用途。
- `主体优先级`
  - 明确用户写入的主体、动作、关系哪些必须保留。
- `画面组织`
  - 约束构图、景别、主体位置、层次与留白方式。
- `镜头与光影`
  - 约束镜头语言、色彩、对比、氛围和质感。
- `传播/版式约束`
  - 强化标签在具体媒介中的使用方式，如标题区、缩略图可读、信息分区等。
- `禁止偏移`
  - 明确不能误入哪些相邻标签风格。

设计重点是“给每个标签建立边界”，而不是简单堆砌更多形容词。

### 4.2 Prompt Merge 规则

prompt merge 层采用“用户主体优先，场景语境增强”的优先级模型：

1. 优先保留用户输入中的主体、动作、关系、时代、地点、情绪核心。
2. 使用场景预设补足影视用途语义、构图方式、镜头语言、光影色彩与传播约束。
3. 当用户描述与场景预设存在冲突时，优先保留用户事实内容，但用场景标签重写表现方式。
4. 输出必须是一段自然语言，不暴露结构字段名，不解释过程。

merge 指令需要显式强调：

- 不得删除用户已明确指定的主体和动作。
- 用户输入较泛时，允许场景标签更强地补足画面组织与风格。
- 用户输入已经具体时，场景标签只负责将其“转换为该标签下应有的表达方式”。
- 最终输出应让人能明显感知这是某个具体标签下的图，而不是泛用图。

### 4.3 一级场景下的重写策略

#### 影视海报创作

- `院线电影主海报`
  - 强调单张主视觉、强叙事冲突、电影级质感、版式留白。
- `网播剧宣传海报（单人/群像）`
  - 强调人物情绪、角色阵列、传播友好感。
- `综艺/纪录片概念海报`
  - 强调节目属性、主题概念与内容识别，而非剧情片张力。

#### IP 角色可视化

- `热门影视IP角色同人形象`
  - 核心是“辨识度 + 二创表达”。
- `未影视化小说IP角色具象化`
  - 核心是“从文本设定落地”。
- `动漫IP角色真人风转绘`
  - 核心是“保留原角色识别特征并真人化”。

#### 影视场景概念设计

- `科幻/奇幻片世界观场景搭建`
  - 强调设定规模、异世界逻辑、空间纵深。
- `古装剧历史还原场景绘制`
  - 强调时代考据、建筑形制与生活痕迹。
- `现代剧生活化场景优化`
  - 强调真实生活感、自然光影与日常情绪。

#### 影视宣发短视频配图

- `剧情高光时刻分镜图`
  - 强调动作瞬间、戏剧冲突、镜头帧感。
- `角色人物关系图`
  - 强调关系表达、角色布局与叙事线索可视化。
- `宣发短视频封面图`
  - 强调抓眼、缩略图识别和信息聚焦。

#### 影视衍生品素材

- `角色Q版表情包`
  - 强调比例简化、表情夸张、传播性强。
- `周边产品图案（手办/文具/服饰）`
  - 强调图案化、可印刷、轮廓清晰。
- `线下主题展打卡点素材`
  - 强调沉浸式场景、空间装置感和可拍照传播。

## 5. 实现边界

本次优化只覆盖“内容层 + 合并规则层”，不扩大到 UI 结构或接口形态。

保留不变：

- `src/App.tsx` 中的一级/二级场景选择结构
- `scene-assisted-prompt -> prompt-merge -> generate-images` 调用链
- `POST /api/generate-images` 接口形态

主要调整文件：

- `server/prompts/*/*.md`
  - 按统一结构重写 15 个二级场景预设
- `server/services/prompt-merge-instruction.ts`
  - 重写系统指令，明确字段优先级和禁止覆盖规则
- `server/services/openai-prompt-merge.ts`
  - 修复拼接文本乱码，改进用户上下文输入结构
- `server/services/gemini-prompt-merge.ts`
  - 修复拼接文本乱码，改进用户上下文输入结构

## 6. 数据流

1. 用户在前端选择一级/二级场景并输入 prompt。
2. 前端保持现有方式请求 `POST /api/generate-images`。
3. 后端读取对应场景预设文件。
4. prompt merge 服务依据新的优先级规则，将“用户主体事实”与“场景标签语义”合并成一段最终 prompt。
5. 生成模型使用最终 prompt 出图。
6. `batch.prompt` 继续保存实际送给模型的最终文本。

## 7. 验证标准

### 7.1 静态验证

- 15 个二级场景预设全部采用统一结构
- 每个文件都有明确的“禁止偏移”
- 同一级场景内的二级标签在用途与视觉语言上有清晰分工

### 7.2 服务层验证

- merge 指令文本无乱码
- provider 侧拼接给模型的上下文无乱码
- merge 输入能够清晰区分“场景预设”和“用户输入”

### 7.3 人工抽样验证

使用相同用户 prompt，分别套入相邻标签，观察最终输出文本是否明显区分，例如：

- `院线电影主海报` vs `宣发短视频封面图`
- `剧情高光时刻分镜图` vs `角色人物关系图`
- `古装剧历史还原场景绘制` vs `科幻/奇幻片世界观场景搭建`

成功标准：

- 能明确看出标签对应的影视用途差异
- 能明确看出构图、镜头、光影、传播语境差异
- 用户原始主体与动作描述仍被保留

## 8. 风险与约束

- 如果只重写预设而不修复 merge 指令乱码，场景贴合度提升会受限。
- 如果 merge 规则过强，可能压过用户输入，违背“平衡约束”目标。
- 如果预设文件结构统一但缺少标签边界，最终仍会出现“每个标签都像通用电影风格 prompt”的问题。
