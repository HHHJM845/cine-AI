# 图片下载能力设计（预览/详情/批量）

## 背景
当前前端页面中多个下载入口没有真实逻辑绑定：
- 预览卡片下载按钮为占位注释。
- 生成结果卡片下载按钮为占位注释。
- 图片详情页“下载素材”按钮未触发下载。
- 资产页批量模式下“下载”按钮未实现。

导致用户点击后“无反应”，不满足基本可用性。

## 目标
在不扩大改动范围的前提下，一次性补齐单图与批量下载能力，确保失败可见、且批量下载不中断。

## 非目标
- 本次不新增后端下载代理接口。
- 本次不重构页面组件拆分。
- 本次不引入复杂通知系统（如全局 Toast 框架）。

## 需求确认
- 文件名规则：`cine-时间戳`（推荐方案）。
- 范围：预览、详情、批量下载一并支持。
- 失败策略：失败提示并继续下载其他图片。

## 方案对比

### 方案 A：`a[download]` 直链
优点：实现最快。
缺点：跨域场景不稳定，常退化为新标签打开，失败难以统计。

### 方案 B（采用）：统一前端下载器（`fetch -> blob -> ObjectURL -> a.click()`）
优点：
- 单图与批量逻辑统一。
- 可捕获失败并给出统计。
- 改动集中在前端，风险可控。
缺点：
- 需补充少量状态管理和错误提示。

### 方案 C：后端下载代理
优点：跨域成功率高、可控性强。
缺点：后端改动大，需处理安全与滥用风险。

## 设计

### 1. 架构与边界
在 `src/App.tsx` 内新增统一下载能力，先不拆分文件：
- `downloadImage(url, filename)`：负责单图下载。
- `downloadImages(items)`：负责批量下载与失败累计。
- 三类入口统一调用同一套下载函数：
  - 预览/探索卡片下载按钮
  - 详情页下载按钮
  - 批量操作栏下载按钮

### 2. 文件名策略
统一命名：`cine-YYYYMMDD-HHmmss-序号.扩展名`
- 扩展名优先从 URL 推断。
- 无法推断时默认 `.png`。

### 3. 数据流
单图：
1. 点击下载按钮。
2. 阻止冒泡，避免触发父级点击。
3. 调用 `downloadImage`。
4. 成功/失败写入提示文案。

批量：
1. 读取 `selectedAssets` 对应图片 URL。
2. 逐条下载（串行），单条失败不终止。
3. 下载结束后展示汇总：`已下载 X 张，失败 Y 张`。

### 4. 错误处理
- `fetch` 非 2xx 时抛错并记录状态码。
- 网络错误/CORS 错误作为失败项统计。
- 每次下载后释放 `ObjectURL`，避免内存泄漏。
- 无选中项触发批量下载时提示“请先选择素材”。

### 5. UI 反馈
新增轻量下载提示文案状态（如 `downloadMessage`），优先复用现有错误展示区附近，不引入新通知组件。

### 6. 验收标准
- 预览卡片下载可用。
- 详情页下载可用。
- 批量下载可用。
- 批量场景下部分失败不影响其余下载。
- 用户可见下载结果提示。

### 7. 验证计划
- `npm run lint`
- `npm run build`
- `npm run dev` 手动验证三条路径（预览、详情、批量）

## 风险与回退
- 风险：跨域源未开放 CORS 时仍可能失败。
- 缓解：提示失败原因并继续其余下载。
- 后续增强：如失败比例高，再评估后端下载代理方案。
