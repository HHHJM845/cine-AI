# 生成请求并行提交与排队预览设计

**日期**: 2026-03-03  
**状态**: 已确认

## 1. 目标

在不改后端接口协议的前提下，让用户在上一批图片尚未生成完成时，仍可继续提交新的生成需求。

本次确认目标：
- 支持连续提交生成需求。
- 同时最多运行 2 个生成任务。
- 超出并行上限的请求自动进入等待队列。
- 预览区以“批次卡 + 逐张出现”方式展示生成进度。
- 部分失败时保留成功图片并显示失败项（`partial_failed`）。

## 2. 范围

### 2.1 本期包含
- 前端生成提交流程从“全局阻塞”改为“队列调度”。
- 预览区新增任务生命周期展示：`queued`、`running`、`completed`、`failed`。
- 同一批次内部先展示占位，再逐张替换为成功/失败结果。

### 2.2 本期不包含
- 后端任务队列与任务持久化。
- 刷新页面后恢复前端队列中的未完成任务。
- 多端共享同一任务队列状态。

## 3. 现状与问题

当前前端 `runGeneration` 使用全局 `isGenerating` 进行互斥，导致请求发出后按钮被禁用，后续点击会直接返回，用户必须等待当前批次完成后才能再次提交。

后端已可独立处理多次 `/api/generate-images` 请求，因此瓶颈主要在前端交互与调度层。

## 4. 方案选型

### 4.1 候选方案
1. 前端本地队列调度（本次采用）
2. 后端任务队列 + 前端轮询
3. 前端队列 + 后端并发兜底

### 4.2 采用方案
采用“前端本地队列调度”，原因：
- 改动集中，落地快。
- 无需改后端协议，风险低。
- 能最快满足“提交后可继续提交”的核心诉求。

## 5. 架构设计

### 5.1 核心职责
- 提交入口：每次点击“生成”都创建一个本地任务 `job`。
- 调度器：维护并发上限 `MAX_CONCURRENT = 2`，从等待队列自动拉起任务。
- 预览投影：将任务状态实时映射到预览区卡片。

### 5.2 状态模型
`UiJob`（前端内存态）建议字段：
- `jobId`
- `prompt`
- `aspectRatio`
- `count`
- `status`：`queued | running | completed | failed`
- `createdAt | startedAt | finishedAt`
- `previewItems`（按 position 固定长度）
- `errorMessage?`

`previewItems` 单项建议字段：
- `position`
- `status`: `loading | success | failed`
- `url?`
- `errorMessage?`

### 5.3 调度流程
1. 用户点击生成。
2. 参数校验通过后，立即插入 `queued` 任务，并初始化 `count` 个 `loading` 占位。
3. 调用 `dispatchNext()`：
- 若 `runningCount < 2`，按 FIFO 启动等待任务。
- 启动后任务状态改为 `running`。
4. 请求完成后：
- 将返回批次映射到对应 `previewItems`。
- 批次状态按返回值转为 `completed` 或 `partial_failed`（UI 显示“部分完成”）。
- 请求级异常时任务置 `failed`。
5. 任务结束时 `runningCount - 1`，再次触发 `dispatchNext()` 继续拉起队列任务。

## 6. 预览区交互设计

采用“批次卡 + 逐张出现”混合模式：
- 提交后立刻出现批次卡。
- 批次卡内先渲染 `count` 个占位位（第 N 张生成中）。
- 后端返回后逐位替换：成功位展示图片，失败位展示错误占位。
- 卡片头部显示：排队中 / 生成中 / 已完成 / 部分完成 / 失败。

## 7. 错误处理与边界

- 请求级错误：当前任务标记为 `failed`，不阻塞后续队列。
- 批次级部分失败：保留成功项，失败位展示错误，整体标记“部分完成”。
- 队列高峰：允许持续入队，显示“排队中 N 个”。
- 执行顺序：FIFO；展示顺序：按提交时间倒序（新任务在顶部）。
- 页面刷新：仅保留后端已落库批次，前端内存队列不恢复（已知折中）。

## 8. 验收标准

1. 连续提交 3 个任务时，状态应为“运行中 2 + 排队中 1”。
2. 任一运行任务完成后，排队头任务自动启动。
3. 生成过程中可继续提交新任务。
4. `partial_failed` 批次在预览区显示成功图 + 失败位。
5. 不再出现“必须等待当前批次完成才能再次提交”的阻塞。
6. `npm run lint`、`npm run test`、`npm run build` 通过。

## 9. 风险与后续

- 风险：前端内存队列在刷新后丢失。
- 后续可演进为“后端任务队列 + 轮询/推送”，实现可恢复与跨端一致性。
