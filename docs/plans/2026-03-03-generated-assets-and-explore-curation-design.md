# 生成资产整合与探索素材扩充设计

## 1. 背景与目标
当前系统存在三类问题：
1. 资产页仍依赖静态 mock 素材，无法只聚焦真实生成结果。  
2. 资产页删除行为尚未实现“前端 + 数据库 + 磁盘文件”的彻底删除闭环。  
3. 探索页图片数量与比例不符合当前需求（需要固定 24 张，比例仅 `16:9` 与 `9:16`）。

本次目标：
1. 资产页仅展示“生成成功图片”，并保留现有交互能力（下载、收藏、批量、详情、删除）。  
2. 删除能力升级为彻底删除（数据库记录 + 磁盘文件 + 前端状态同步）。  
3. 探索页扩充为 24 张本地静态素材，且仅使用 `16:9`、`9:16`。  
4. 探索页素材与资产页数据不互通（用户明确要求）。

## 2. 范围与非目标
### 2.1 范围
- 前端：`src/App.tsx`、探索素材源、资产列表数据源、删除交互。  
- 前端服务层：新增/扩展删除 API 调用。  
- 后端：新增删除接口与仓储/存储层删除能力。  
- 测试：前后端针对删除与数据映射补充测试。

### 2.2 非目标
- 不新增探索页上传入口（本次为一次性导入图片）。  
- 不将探索图片同步进资产页。  
- 不引入新 UI 框架或第三方瀑布流库。  
- 不改动生成模型与生成参数协议。

## 3. 方案选型
采用“本地文件驱动探索素材 + 资产页仅生成成功图 + 后端彻底删除接口”的组合方案。

选择依据：
1. 满足用户对数据边界的明确要求（探索与资产解耦）。  
2. 删除链路可验证、可追溯，避免前端假删除。  
3. 改动范围可控，能与现有生成批次结构复用。

## 4. 架构设计
### 4.1 探索页（静态素材）
- 新增目录：`public/explore/`。  
- 探索素材元数据固定 24 条，字段包含：`id/url/prompt/ratio/model/seed`。  
- `ratio` 仅允许 `16:9` 与 `9:16`。  
- 保留现有搜索与详情弹层逻辑。

### 4.2 资产页（生成结果）
- 移除 `ASSET_IMAGES` 等静态数据在资产页的参与。  
- 由 `generationBatches` 派生 `assetItems`：
  - 仅纳入 `status === success && url 存在` 的项；
  - 派生字段：`id/url/prompt/ratio/model/timestamp`。  
- 现有交互（收藏、批量选择、日期筛选、下载、详情）全部作用于 `assetItems`。

### 4.3 删除闭环
- 后端新增删除接口（支持单个与批量）。  
- 删除处理步骤：
  1. 根据 item id 查出 `image_path`；
  2. 删除磁盘文件（不存在时按幂等成功处理）；
  3. 删除 `generation_items` 记录；
  4. 若 batch 无剩余 item，删除 `generation_batches`。  
- 前端删除成功后同步更新 `generationBatches`，确保资产页即时刷新。

## 5. 数据流与交互
### 5.1 探索页数据流
`EXPLORE_IMAGES(24 条本地元数据) -> 搜索过滤 -> 卡片渲染 -> 详情/下载`

### 5.2 资产页数据流
`generationBatches(API) -> success items 扁平化 -> sortedAssets 过滤/排序 -> 资产渲染`

### 5.3 批量删除交互
`selectedAssets(id集合) -> 删除 API -> 返回成功/失败明细 -> 前端只移除成功项并提示统计`

## 6. 错误处理策略
1. 删除接口失败：前端不移除数据，toast 展示失败原因。  
2. 文件不存在：后端继续删除 DB 记录并视为成功（幂等）。  
3. 批量部分失败：返回分项结果，前端部分更新并提示“成功 X / 失败 Y”。

## 7. 测试与验收
### 7.1 自动化验证
1. `npm run lint` 通过。  
2. `npm run test` 通过。  
3. `npm run build` 通过。

### 7.2 手动验收
1. 探索页显示 24 张图，比例仅 `16:9`/`9:16`。  
2. 资产页只显示生成成功图片，不再出现旧静态素材。  
3. 资产页下载/收藏/批量/详情功能正常。  
4. 删除后刷新页面仍不出现，且磁盘对应文件已删除。

## 8. 风险与缓解
1. 风险：历史数据中部分 item 无 `image_path`。  
缓解：删除接口对空路径做兼容，仅清理 DB。  
2. 风险：批量删除时前端状态不同步。  
缓解：删除成功后以返回的成功 id 为准更新本地状态。  
3. 风险：探索图片文件命名不规范导致 404。  
缓解：固定命名约定并在导入后统一自检一次。
